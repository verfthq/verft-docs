# Getting Started with Verft

## Overview

Verft is a self-hosted control plane for running parallel AI coding agents across your Git repositories.

Use this guide to start Verft locally, sign in, connect provider credentials, add a repository, and create your first task.

## Before You Begin

You need:

- Docker
- Docker Compose
- Access to the Verft repository
- GitHub credentials for the repositories you want Verft to access
- OpenAI/Codex and/or Anthropic/Claude credentials

Docker is required for the Verft app stack and the agent runtime containers.

## Steps

1. Clone the Verft repository:

   ```bash
   git clone git@github.com:verfthq/verft.git
   cd verft
   ```

2. Create a local environment file:

   ```bash
   cp .env.example .env
   ```

3. Review the bootstrap admin credentials in `.env`.

   Verft uses these credentials only when the first admin user is created. Change the default values before exposing Verft outside a local environment.

4. Initialize the Docker stack and agent runtime image:

   ```bash
   ./verft init
   ```

5. Start Verft:

   ```bash
   ./verft start
   ```

6. Open Verft in your browser:

   ```text
   http://localhost:3217/login
   ```

7. Sign in with the bootstrap admin credentials from `.env`.

   The default credentials from `.env.example` are:

   | Field | Default value |
   | --- | --- |
   | Email | `admin@verft.local` |
   | Password | `admin123!` |

   Change these values before exposing Verft outside a local environment.

8. Open **Settings**.

9. Add GitHub credentials.

10. Add OpenAI/Codex credentials and/or Anthropic/Claude credentials.

11. Open **Repositories**.

12. Add the Git repository you want agents to work on.

13. Open **Tasks**.

14. Create a build task or an ask task.

15. Select the repository, enter the prompt, choose provider and model settings if needed, and start the task.

## Result

Verft starts locally at `http://localhost:3217/login`.

After setup, you can create tasks for the connected repository:

- **Build tasks** ask an agent to modify files in an isolated workspace.
- **Ask tasks** ask an agent to inspect the repository and answer without writing files.

Open the task detail page to review task status, streamed logs, agent messages, terminal output, checkpoints, pending diffs, and Git operations.

Build task changes do not have to move forward automatically. Review the result before applying, rejecting, reverting, pushing, or merging changes.

## Troubleshooting

- Problem: Verft does not open at `http://localhost:3217/login`.
  - Solution: Confirm the stack is running with `./verft start`, then check the health endpoint:

    ```bash
    curl -fsS http://localhost:3217/api/health
    ```

- Problem: The first admin sign-in does not work.
  - Solution: Check the bootstrap admin values in `.env`. These values come from `.env.example` unless you changed them before initialization.

- Problem: A task cannot access the expected repository or credentials.
  - Solution: Open **Settings** and confirm the required GitHub and provider credentials are configured. Then open **Repositories** and confirm the repository is added with the expected configuration.

- Problem: Verft is deployed to `example.com` or `subdomain.example.com`, but browser requests fail because of CORS.
  - Solution: Set `CORS_ORIGIN` in `.env` to the exact public origin users open in the browser, including the protocol:

    ```bash
    CORS_ORIGIN=https://example.com
    ```

    For a subdomain, use that subdomain:

    ```bash
    CORS_ORIGIN=https://subdomain.example.com
    ```

    Restart Verft after changing `.env`.

- Problem: You want to stop the local Verft stack.
  - Solution: Run:

    ```bash
    ./verft stop
    ```
